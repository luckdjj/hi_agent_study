▶ 1.1 什么是智能体？

核心定义

在人工智能领域，智能体（Agent）被定义为：

任何能够通过传感器（Sensors）感知其所处环境（Environment），并自主地通过执行器（Actuators）采取行动（Action）以达成特定目标的实体。

四个基本要素

  1. 环境（Environment）

    智能体所处的外部世界。对于自动驾驶汽车，环境是动态变化的道路交通；对于一个交易算法，环境则是瞬息万变的金融市场。

  2. 传感器（Sensors）

    智能体通过传感器持续地感知环境状态。摄像头、麦克风、雷达或各类 API 返回的数据流，都是其感知能力的延伸。

  3. 执行器（Actuators）

    智能体通过执行器来改变环境的状态。执行器可以是物理设备（如机械臂、方向盘）或虚拟工具（如执行一段代码、调用一个服务）。

  4. 自主性（Autonomy）

    真正赋予智能体"智能"的核心。智能体并非只是被动响应外部刺激或严格执行预设指令的程序，它能够基于其感知和内部状态进行独立决策，以达成其设计目标。

💡 关键理解：

这种从感知到行动的闭环，构成了所有智能体行为的基础。智能体不是孤立的，而是与环境持续交互的动态系统。

▶ 1.1.2 大语言模型驱动的新范式

范式转变

以 GPT（Generative Pre-trained Transformer）为代表的大语言模型的出现，正在显著改变智能体的构建方法与能力边界。

传统智能体 vs LLM 智能体

维度 传统智能体 LLM 智能体
核心引擎 显式编程规则 隐式学习的神经网络
知识来源 人工构建知识库 海量数据预训练
交互方式 结构化指令 自然语言对话

LLM 智能体的三大核心能力

1. 规划与推理（Planning & Reasoning）

智能体首先会将高层级目标分解为一系列逻辑子任务。例如，当接收到"规划一次厦门之旅"这样的模糊指令时，它会自动分解为：

[确认出行偏好] → [查询目的地信息] → [制定行程草案] → [预订票务住宿]

这是一个内在的、由模型驱动的规划过程。

2. 工具使用（Tool Use）

在执行规划时，智能体识别到信息缺口，会主动调用外部工具来补全。例如：

  • 调用天气查询接口获取实时天气
  • 基于"预报有雨"这一信息，在后续规划中倾向于推荐室内活动

3. 动态修正（Dynamic Adjustment）

在交互过程中，智能体会将用户的反馈（如"这家酒店超出预算"）视为新的约束，并据此调整后续的行动，重新搜索并推荐符合新要求的选项。

✅ 核心转变：

我们正从开发专用自动化工具转向构建能自主解决问题的系统。核心不再是编写代码，而是引导一个通用的"大脑"去规划、行动和学习。

▶ 1.2.2 智能体的运行机制

智能体循环（Agent Loop）

智能体并非一次性完成任务，而是通过一个持续的循环与环境进行交互。这个核心机制被称为智能体循环（Agent Loop），它描述了智能体与环境之间的动态交互过程。

循环的四个阶段

1. 感知（Perception）

这是循环的起点。智能体通过其传感器（例如，API 的监听端口、用户输入接口）接收来自环境的输入信息。

  • 这些信息，即观察（Observation）
  • 既可以是用户的初始指令
  • 也可以是上一步行动所导致的环境状态变化反馈

2. 思考（Thought）

接收到观察信息后，智能体进入其核心决策阶段。对于 LLM 智能体而言，这通常是由大语言模型驱动的内部推理过程。

"思考"阶段可进一步细分为两个关键环节：

  • 规划（Planning）

    智能体基于当前的观察和其内部记忆，更新对任务和环境的理解，并制定或调整一个行动计划。这可能涉及将复杂目标分解为一系列更具体的子任务。

  • 工具选择（Tool Selection）

    根据当前计划，智能体从其可用的工具库中，选择最适合执行下一步骤的工具，并确定调用该工具所需的具体参数。

3. 行动（Action）

决策完成后，智能体通过其执行器（Actuators）执行具体的行动。这通常表现为调用一个选定的工具（如代码解释器、搜索引擎 API），从而对环境施加影响，意图改变环境的状态。

4. 观察（Observation）

行动并非循环的终点。智能体的行动会引起环境（Environment）的状态变化（State Change），环境随即会产生一个新的观察（Observation）作为结果反馈。

这个新的观察又会在下一轮循环中被智能体的感知系统捕获，形成一个持续的闭环。

🔄 循环的本质：

智能体正是通过不断重复这一循环，逐步推进任务，从初始状态向目标状态演进。这个循环会一直持续，直到达成目标或满足终止条件。

📝 实例：智能旅行助手
  1. 感知：接收用户请求"查询北京天气并推荐景点"
  2. 思考：规划分两步 → 先查天气，再推荐景点
  3. 行动：调用 get_weather("北京")
  4. 观察：获得"晴天，25度"
  5. 进入下一轮循环...
▶ 1.2.3 智能体的感知与行动

Thought-Action-Observation 交互协议

在工程实践中，为了让 LLM 能够有效驱动智能体循环，我们需要一套明确的交互协议（Interaction Protocol）来规范其与环境之间的信息交换。

协议的三个核心部分

1. Thought（思考）

这是智能体内部决策的"快照"。它以自然语言形式阐述了：

  • 智能体如何分析当前情境
  • 回顾上一步的观察结果
  • 进行自我反思与问题分解
  • 最终规划出下一步的具体行动

2. Action（行动）

这是智能体基于思考后，决定对环境施加的具体操作，通常以函数调用的形式表示。

3. Observation（观察）

环境对行动的反馈。原始的机器可读数据（如 JSON）会被处理并封装成简洁、清晰的自然语言文本。

完整交互示例

Thought: 用户想知道北京的天气。我需要调用天气查询工具。
Action: get_weather("北京")
Observation: 北京当前天气为晴，气温25摄氏度，微风。

工作流程详解

  1. LLM 生成输出

    智能体生成包含 Thought 和 Action 的结构化文本

  2. 解析器捕获指令

    外部的解析器（Parser）会捕捉到 Action 字段，并调用相应的函数

  3. 执行并获取结果

    例如，get_weather函数返回一个包含详细天气数据的 JSON 对象

  4. 封装为 Observation

    感知系统将原始输出处理成自然语言文本

  5. 反馈给智能体

    这段 Observation 文本会被反馈给智能体，作为下一轮循环的输入

✅ 核心价值：

通过这个由 Thought、Action、Observation 构成的严谨循环，LLM 智能体得以将内部的语言推理能力，与外部环境的真实信息和工具操作能力有效地结合起来。

AGENT_SYSTEM_PROMPT = """

# 输出格式要求:

- Action必须在同一行，不要换行

工具 1：查询真实天气

使用免费的天气查询服务 wttr.in，它能以 JSON 格式返回指定城市的天气数据。

import requests

def get_weather(city: str) -> str:
    """通过调用 wttr.in API 查询真实的天气信息。"""
    url = f"https://wttr.in/{city}?format=j1"

    try:
        response = requests.get(url)
        response.raise_for_status()
        data = response.json()

        current_condition = data['current_condition'][0]
        weather_desc = current_condition['weatherDesc'][0]['value']
        temp_c = current_condition['temp_C']

        return f"{city}当前天气:{weather_desc}，气温{temp_c}摄氏度"

    except requests.exceptions.RequestException as e:
        return f"错误:查询天气时遇到网络问题 - {e}"
    except (KeyError, IndexError) as e:
        return f"错误:解析天气数据失败，可能是城市名称无效 - {e}"

工具 2：搜索并推荐旅游景点

使用 Tavily Search API 根据城市和天气状况搜索合适的景点。

import os
from tavily import TavilyClient

def get_attraction(city: str, weather: str) -> str:
    """根据城市和天气，使用Tavily Search API搜索并返回景点推荐。"""
    api_key = os.environ.get("TAVILY_API_KEY")
    if not api_key:
        return "错误:未配置TAVILY_API_KEY环境变量。"

    tavily = TavilyClient(api_key=api_key)
    query = f"'{city}' 在'{weather}'天气下最值得去的旅游景点推荐及理由"

    try:
        response = tavily.search(query=query, search_depth="basic", include_answer=True)

        if response.get("answer"):
            return response["answer"]

        formatted_results = []
        for result in response.get("results", []):
            formatted_results.append(f"- {result['title']}: {result['content']}")

        if not formatted_results:
            return "抱歉，没有找到相关的旅游景点推荐。"

        return "根据搜索，为您找到以下信息:\n" + "\n".join(formatted_results)

    except Exception as e:
        return f"错误:执行Tavily搜索时出现问题 - {e}"

# 工具字典
available_tools = {
    "get_weather": get_weather,
    "get_attraction": get_attraction,
}

1.3.2 接入大语言模型

实现一个通用的客户端，可以连接到任何兼容 OpenAI 接口规范的 LLM 服务。

from openai import OpenAI

class OpenAICompatibleClient:
    """用于调用任何兼容OpenAI接口的LLM服务的客户端。"""
    def __init__(self, model: str, api_key: str, base_url: str):
        self.model = model
        self.client = OpenAI(api_key=api_key, base_url=base_url)

    def generate(self, prompt: str, system_prompt: str) -> str:
        """调用LLM API来生成回应。"""
        print("正在调用大语言模型...")
        try:
            messages = [
                {'role': 'system', 'content': system_prompt},
                {'role': 'user', 'content': prompt}
            ]
            response = self.client.chat.completions.create(
                model=self.model,
                messages=messages,
                stream=False
            )
            answer = response.choices[0].message.content
            print("大语言模型响应成功。")
            return answer
        except Exception as e:
            print(f"调用LLM API时发生错误: {e}")
            return "错误:调用语言模型服务时出错。"

1.3.3 执行行动循环

主循环整合所有组件，通过格式化后的 Prompt 驱动 LLM 进行决策。

import re

# 配置LLM客户端
API_KEY = "YOUR_API_KEY"
BASE_URL = "YOUR_BASE_URL"
MODEL_ID = "YOUR_MODEL_ID"
os.environ['TAVILY_API_KEY'] = "YOUR_TAVILY_API_KEY"

llm = OpenAICompatibleClient(
    model=MODEL_ID,
    api_key=API_KEY,
    base_url=BASE_URL
)

# 初始化
user_prompt = "你好，请帮我查询一下今天北京的天气，然后根据天气推荐一个合适的旅游景点。"
prompt_history = [f"用户请求: {user_prompt}"]

print(f"用户输入: {user_prompt}\n" + "="*40)

# 运行主循环
for i in range(5):  # 设置最大循环次数
    print(f"--- 循环 {i+1} ---\n")

    # 构建Prompt
    full_prompt = "\n".join(prompt_history)

    # 调用LLM进行思考
    llm_output = llm.generate(full_prompt, system_prompt=AGENT_SYSTEM_PROMPT)
    print(f"模型输出:\n{llm_output}\n")
    prompt_history.append(llm_output)

    # 解析并执行行动
    action_match = re.search(r"Action: (.*)", llm_output, re.DOTALL)
    if not action_match:
        observation = "错误: 未能解析到 Action 字段。"
        observation_str = f"Observation: {observation}"
        print(f"{observation_str}\n" + "="*40)
        prompt_history.append(observation_str)
        continue

    action_str = action_match.group(1).strip()

    if action_str.startswith("Finish"):
        final_answer = re.match(r"Finish\[(.*)\]", action_str).group(1)
        print(f"任务完成，最终答案: {final_answer}")
        break

    tool_name = re.search(r"(\w+)\(", action_str).group(1)
    args_str = re.search(r"\((.*)\)", action_str).group(1)
    kwargs = dict(re.findall(r'(\w+)="([^"]*)"', args_str))

    if tool_name in available_tools:
        observation = available_tools[tool_name](**kwargs)
    else:
        observation = f"错误:未定义的工具 '{tool_name}'"

    # 记录观察结果
    observation_str = f"Observation: {observation}"
    print(f"{observation_str}\n" + "="*40)
    prompt_history.append(observation_str)

P1 进阶理解 理解智能体生态和选型的背景知识
▶ 1.1.1 传统视角下的智能体

演进路径概述

在大语言模型热潮出现之前，人工智能的先驱们已经对"智能体"这一概念进行了数十年的探索与构建。这些传统智能体经历了一条从简单到复杂、从被动反应到主动学习的清晰演进路线。

1. 反射智能体（Simple Reflex Agent）

这是演进的起点，结构最简单的智能体类型。

  • 决策核心：由工程师明确设计的"条件-动作"规则
  • 工作方式：完全依赖于当前的感知输入，不具备记忆或预测能力
  • 经典案例：自动恒温器
    • 若传感器感知的室温高于设定值 → 启动制冷系统
    • 若室温低于设定值 → 启动加热系统
  • 优势：可靠且高效，响应速度快
  • 局限：像一种数字化的本能，无法应对需要理解上下文的复杂任务

2. 基于模型的反射智能体（Model-Based Reflex Agent）

为了解决"当前状态不足以作为决策全部依据"的问题，引入了"状态"的概念。

  • 核心特征：拥有一个内部的世界模型（World Model）
  • 世界模型的作用：追踪和理解环境中那些无法被直接感知的方面
  • 回答的问题："世界现在是什么样子的？"
  • 经典案例：隧道中的自动驾驶汽车
    • 即便摄像头暂时无法感知到前方的车辆
    • 内部模型依然会维持对那辆车存在、速度和预估位置的判断
  • 进步意义：拥有了初级的"记忆"，决策不再仅仅依赖于瞬时感知

3. 基于目标的智能体（Goal-Based Agent）

仅仅理解世界还不够，智能体需要有明确的目标。

  • 核心特征：行为不再是被动地对环境做出反应，而是主动地、有预见性地选择行动
  • 回答的问题："我应该做什么才能达成目标？"
  • 核心能力：对未来的考量与规划（Planning）
  • 经典案例：GPS 导航系统
    • 目标：到达公司
    • 基于地图数据（世界模型）
    • 通过搜索算法（如 A* 算法）规划出一条最优路径

4. 基于效用的智能体（Utility-Based Agent）

现实世界的目标往往不是单一的，需要在多个目标之间权衡。

  • 核心机制：为每一个可能的世界状态都赋予一个效用值
  • 效用值含义：代表满意度的高低
  • 核心目标：最大化期望效用
  • 回答的问题："哪种行为能为我带来最满意的结果？"
  • 实际应用：导航系统不仅考虑到达目的地，还要权衡：
    • 时间最短
    • 路程最省油
    • 避开拥堵
  • 进步意义：决策更接近人类的理性选择

5. 学习型智能体（Learning Agent）

前面讨论的智能体虽然功能日益复杂，但其核心决策逻辑依然依赖于人类设计师的先验知识。学习型智能体则能通过与环境的互动自主学习。

  • 核心思想：通过与环境的互动自主学习
  • 实现路径：强化学习（Reinforcement Learning, RL）
  • 组成部分：
    • 性能元件：前面讨论的各类智能体
    • 学习元件：观察性能元件的行动结果，不断修正决策策略
  • 经典案例：AlphaGo Zero
    • 开始时只是随机落子
    • 通过大量的自我对弈
    • 逐渐发现哪些棋路更有可能导向最终的胜利
    • 发现了许多超越人类既有知识的有效策略
📈 演进总结：

从简单的恒温器 → 拥有内部模型的汽车 → 能够规划路线的导航 → 懂得权衡利弊的决策者 → 可以通过经验自我进化的学习者。这条演进之路，展示了传统人工智能在构建机器智能的道路上所经历的发展脉络。

▶ 1.1.3 智能体的类型

分类维度概述

智能体可以从三个互补的维度进行分类，每个维度揭示了智能体设计的不同方面。

维度一：基于决策时间与反应性

这个视角关注智能体是在接收到信息后立即行动，还是会经过深思熟虑的规划再行动。揭示了智能体设计中一个核心权衡：追求速度的反应性（Reactivity）与追求最优解的规划性（Deliberation）之间的平衡。

1. 反应式智能体（Reactive Agents）

  • 特征：对环境刺激做出近乎即时的响应，决策延迟极低
  • 决策方式：遵循从感知到行动的直接映射，不进行或只进行极少的未来规划
  • 优势：
    • 速度快
    • 计算开销低
    • 适合需要快速决策的动态环境
  • 劣势：
    • "短视"，缺乏长远规划
    • 容易陷入局部最优
    • 难以完成需要多步骤协调的复杂任务
  • 典型应用：
    • 车辆的安全气囊系统（必须在碰撞发生的毫秒内做出反应）
    • 高频交易机器人（捕捉稍纵即逝的市场机会）

2. 规划式智能体（Deliberative Agents）

  • 特征：在行动前会进行复杂的思考和规划
  • 决策方式：
    • 利用内部的世界模型
    • 系统地探索未来的各种可能性
    • 评估不同行动序列的后果
    • 找到一条能够达成目标的最佳路径
  • 类比：像一位棋手，不会只看眼前的一步，而是会预想对手可能的应对，并规划出后续几步甚至十几步的棋路
  • 优势：
    • 决策的战略性和远见
    • 能够处理复杂的、需要长远眼光的任务
  • 劣势：
    • 高昂的时间和计算成本
    • 在瞬息万变的环境中可能错过最佳时机
  • 典型应用：
    • 制定商业计划
    • 规划长途旅行

3. 混合式智能体（Hybrid Agents）

  • 设计目标：结合反应式和规划式的优点，实现反应与规划的平衡
  • 经典架构：分层设计
    • 底层：快速的反应模块，处理紧急情况和基本动作
    • 高层：审慎的规划模块，负责制定长远目标
  • 现代 LLM 智能体的混合模式：
    • 在"思考-行动-观察"的循环中运作
    • 规划（Reasoning）：在"思考"阶段，LLM 分析当前状况，规划下一步行动
    • 反应（Acting & Observing）：在"行动"和"观察"阶段，与外部工具或环境交互，立即获得反馈
    • 将一个需要长远规划的宏大任务，分解为一系列"规划-反应"的微循环

维度二：基于知识表示

这是一个更根本的分类维度，探究智能体用以决策的知识，究竟是以何种形式存于其"思想"之中。这个问题是人工智能领域一场持续半个多世纪的辩论核心。

1. 符号主义 AI（Symbolic AI）

  • 核心信念：智能源于对符号的逻辑操作
  • 符号：人类可读的实体（如词语、概念）
  • 操作：遵循严格的逻辑规则
  • 类比：像一位一丝不苟的图书管理员，将世界知识整理为清晰的规则库和知识图谱
  • 优势：
    • 透明和可解释
    • 推理步骤明确
    • 决策过程可以被完整追溯
    • 在金融、医疗等高风险领域至关重要
  • 劣势：
    • 脆弱性：依赖于一个完备的规则体系
    • 知识获取瓶颈：任何未被覆盖的新情况都可能导致系统失灵
    • 在充满模糊和例外的现实世界中难以应对

2. 亚符号主义 AI（Sub-symbolic AI）

  • 别名：连接主义
  • 核心特征：知识并非显式的规则，而是内隐地分布在一个由大量神经元组成的复杂网络中
  • 知识形式：从海量数据中学习到的统计模式
  • 代表技术：神经网络和深度学习
  • 类比：像一个牙牙学语的孩童
    • 不是通过学习"猫有四条腿、毛茸茸、会喵喵叫"这样的规则来认识猫
    • 而是在看过成千上万张猫的图片后，大脑中的神经网络能辨识出"猫"这个概念的视觉模式
  • 优势：
    • 强大的模式识别能力
    • 对噪声数据的鲁棒性
    • 能够轻松处理图像、声音等非结构化数据
  • 劣势：
    • 不透明性：通常被视为一个黑箱（Black Box）
    • 无法给出合乎逻辑的解释
    • 在纯粹的逻辑推理任务上表现不佳
    • 有时会产生看似合理却事实错误的幻觉

3. 神经符号主义 AI（Neuro-Symbolic AI）

  • 设计目标：融合符号主义和亚符号主义的优点
  • 核心理念：创造出一个既能像神经网络一样从数据中学习，又能像符号系统一样进行逻辑推理的混合智能体
  • 理论类比：卡尼曼的双系统理论（《思考，快与慢》）
    • 系统 1：快速、凭直觉、并行的思维模式 ≈ 亚符号主义 AI 的模式识别能力
    • 系统 2：缓慢、有条理、基于逻辑的审慎思维 ≈ 符号主义 AI 的推理过程
  • 实践范例：大语言模型驱动的智能体
    • 内核是一个巨大的神经网络（模式识别和语言生成能力）
    • 生成一系列结构化的中间步骤（思想、计划或 API 调用）
    • 实现了感知与认知、直觉与理性的初步融合
🎯 分类的意义：

理解这些分类维度，有助于我们在设计智能体时做出明智的架构选择，根据具体应用场景选择合适的智能体类型。
